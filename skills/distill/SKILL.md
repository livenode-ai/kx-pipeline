# /distill — 蒸留

セッションの成果を記録・保存する。ナレッジのライフサイクルにおける「記録→保存」工程。
作成されたハンドオーバーは 00_Inbox/ に着地し、次回の /kx で notes/ に精錬される。

Optional: add mcp__todoist__* tools if using Todoist integration

---

## 実行

以下のキーワードで発動：
- 「蒸留して」
- 「HO作って」
- 「ハンドオーバー」
- 「/distill」

---

## Step 1: ハンドオーバー作成

今回のセッション内容を記録し、`00_Inbox/` に保存する。

### ファイル名
`00_Inbox/YYYY-MM-DD_[テーマ]_HO.md`

### 構造

```markdown
# [プロジェクト名] ハンドオーバー

日付: YYYY-MM-DD
ステータス: 進行中 / 完了 / 保留

## 意思決定とその理由
- [何を決めたか] — [なぜそう判断したか]

## 方針転換（あれば）
- 変更前 → 変更後 → 理由

## 固有名詞と数字
- [人名・組織名・金額・期日・数量は省略しない]

## 未解決の論点
- [結論が出ていない問題]

## 次のアクション
- [ ] [アクション内容] — 担当: user / AI
```

### 必ず残すもの（絶対に落とさない）

- **意思決定とその理由**: 「何を」だけでなく「なぜ」をセットで
- **固有名詞と数字**: 人名・組織名・金額・期日は省略しない
- **独自用語・フレームワーク**: 初出時は定義を併記
- **未解決の論点**: 結論が出ていない問題こそ次に必要
- **方針転換**: 変更前→変更後→理由の3点セット

### 省略してよいもの

- 作業手順の詳細（コマンド、操作ステップ）
- 途中の試行錯誤や修正プロセス
- AIが自力で再生成できる一般的な技術説明
- 挨拶・雑談・脱線

---

## Step 2: BRIEFING.md 差分更新

該当プロジェクトの `01_Projects/[プロジェクト]/BRIEFING.md` を差分更新する。

- 全体を書き直さない。今回のセッションで変わった部分だけ反映
- 可能な限り元の会話ログから蒸留する（要約の要約は劣化する）
- 基準: 「このプロジェクトに今日から参加する人が読んで全体像を把握できる」
- BRIEFING.md が存在しなければ新規作成

---

## Step 3: VISION.md 差分更新

以下があれば VISION.md に追記する:
- 新しい構想・ビジョン
- 未解決の問い
- 成長の記録（新しくできるようになったこと）
- 思想的飛躍

なければスキップ。

---

## Step 3.5: AESTHETIC.md 追記（あれば）

AIが出した提案に対してユーザーが「ここ違う」「これは俺らしい」と感じた差分があれば、AESTHETIC.md の YES/NO パターンに1行追記する。

なければスキップ。

---

## Step 4: MEMORY.md 差分更新

以下があれば MEMORY.md に追記する:
- 進行中プロジェクトの状態変化
- 新しい意思決定
- 技術スタック変更

なければスキップ。

---

## Step 5: Todoist タスク登録

If Todoist integration is enabled (see livenode-kx.config.yaml):

Step 1 の「次のアクション」に書いた項目を Todoist に登録する。

- `todoist_task_create` or `todoist_task_quick_add` を使う
- プロジェクト名が既存の Todoist プロジェクトと一致すればそこに、なければ Inbox に
- 期日がわかるものは `due_string` に設定

---

## 完了レポート

```
## Distill Complete

- HO: 00_Inbox/YYYY-MM-DD_[テーマ]_HO.md
- BRIEFING: [更新した / スキップ]
- VISION: [更新した / スキップ]
- MEMORY: [更新した / スキップ]
- Todoist: N件登録
```
