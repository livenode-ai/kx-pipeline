# Architecture

## The 3-Layer Model

Most AI memory tools connect your knowledge directly to the LLM:

```
[Your Notes / External KB] ──────────────────────────→ [LLM]
                            semantic distance retrieval   generic output
```

This works. But the output is calibrated to external averages — the best synthesis of information the LLM has seen. It's not calibrated to *you*.

livenode-kx adds the missing judgment layer:

```
┌──────────────────────────────┐
│  Your Notes / External KB    │
│  (Obsidian, Logseq, files)   │
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│   Judgment Layer             │  ← livenode-kx lives here
│                              │
│  • Noetic Cell claims        │
│    (structured, staged)      │
│  • Decision history          │
│    (BRIEFING.md + /distill)  │
│  • YES/NO patterns           │
│    (AESTHETIC.md)            │
│  • Active project context    │
│    (CLAIMS_INDEX)            │
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│   LLM Output                 │
│   calibrated to your         │
│   reasoning, not averages    │
└──────────────────────────────┘
```

**The analogy:** Most AI tools give you better glasses. livenode-kx gives you prescription lenses — same world, but calibrated to your specific vision. More data doesn't improve output if it's not filtered through how you see.

## Components

### skills/ — The Pipeline

| Skill | Phase | What it does |
|---|---|---|
| `/kx` | Capture → Knowledge | 5-phase: scan → mine → connect → revisit → index |
| `/distill` | Session → Memory | handover → BRIEFING update → AESTHETIC calibration |
| `/health` | Maintenance | 4 diagnostics: schema, orphans, links, map balance |
| `/ref` | Retrieval | CLAIMS_INDEX → staged reading → targeted deep read |

### schemas/ — The Structure

- **noetic-cell.md** — The 5-section note format. All insight/decision/synthesis notes use it.
- **note-frontmatter.yaml** — Required fields: `description`, `type`, `domain`, `created`, `topics`. Optional: `domain_weight`.
- **domain-map.md** — Map of Content (MOC) files that organize notes by domain.
- **aesthetic-layer.md** — Design spec for AESTHETIC.md (judgment calibration log).

### example-vault/ — Reference Implementation

A working vault with:
- A raw inbox capture (`00_Inbox/example-raw-capture.md`)
- A processed Noetic Cell note (`notes/`)
- A sample AESTHETIC.md
- A sample `self/identity.md`

Run `python3 tests/validate_schema.py example-vault/notes/` to see it pass.

### livenode-kx.config.yaml — User Configuration

The single file users customize. Controls vault paths, domain names, and optional integrations (Todoist, Obsidian MCP). Skills read from this file — no hardcoded paths anywhere.

## Data Flow

```
Voice memo / quick note
        │
        ▼ (manual: drop into 00_Inbox/)
  00_Inbox/YYYY-MM-DD_topic.md
        │
        ▼ (/kx)
  Phase 1: Scan  → list all inbox files
  Phase 2: Mine  → extract atomic notes → notes/*.md
  Phase 3: Connect → link to existing notes (bidirectional)
  Phase 4: Revisit → update old notes with new connections
  Phase 5: Index  → regenerate CLAIMS_INDEX.md
        │
        ▼
  notes/ (structured knowledge graph)
  notes/CLAIMS_INDEX.md (generated by /kx Phase 5: Index)
        │
        ▼ (end of session: /distill)
  00_Inbox/YYYY-MM-DD_topic_HO.md (handover)
  BRIEFING.md (updated; path is configurable in livenode-kx.config.yaml)
  AESTHETIC.md (judgment calibration log, if delta found)
        │
        ▼ (next /kx run)
  handover → notes/ (persisted as knowledge)
```

## Compatibility

livenode-kx uses local file operations by default (Read, Write, Edit, Bash). No external services required.

| Integration | Config flag | Notes |
|---|---|---|
| Local files only | `obsidian_mcp: false` (default) | Works everywhere |
| Obsidian MCP | `obsidian_mcp: true` | Faster vault operations |
| Todoist | `todoist: false` (default) | Optional task registration in /distill |
| Cloudflare Artifacts | — | Planned (not yet implemented) |
